# Father and Son Skills

Father and Son Skills 是一个入口技能。它不直接实现业务，而是负责路由、复用判断、特化分流、草稿管理和生效控制。

它的目标不是把答案一次写完，而是先生成草稿，再询问用户是否要载入生效池。用户也可以显式指定某个子 skill，要求直接调用已有版本，或者创建一个新的指定子 skill 草稿。

## 它是做什么的

这个 skill 主要做五件事：

- 路由：判断当前任务该看现成内容还是草稿。
- 约束：保证只有生效池里的内容可以被正式加载和引用。
- 复用：优先复用 `active/` 中已有内容，避免重复生成。
- 显式调用：当用户明确指定某个子 skill 时，直接优先查找并调用它。
- 显式创建：当用户明确指定要创建某个子 skill 时，生成该子 skill 的草稿。

## 什么时候用

在下面这些场景使用它：

- 你需要通过 Father and Son Skills 入口处理工作。
- 你需要读取 `.father-and-son-skills` 状态目录。
- 你要判断某个内容能不能复用、能不能载入生效池。
- 用户明确说要调用某个子 skill。
- 用户明确说要创建某个子 skill。
- 你要把新内容先写成草稿，再询问用户是否要载入生效池。

## 什么时候不用

不要在下面这些场景使用它：

- 你要直接实现业务逻辑、页面、接口、脚本或算法。
- 你已经明确知道要改的是某个具体文件，不需要状态流转。
- 你只是做普通开发，和 `.father-and-son-skills` 目录无关。
- 你想把草稿以外的内容直接当正式内容加载。

## 读取和加载规则

### 基本规则

- 先读 `.father-and-son-skills/README.md`。
- 只加载 `.father-and-son-skills/active/`，这是唯一正式可用的内容。
- `.father-and-son-skills/drafts/` 只作为草稿中间态，不加载。
- 任何内容进入 `active/` 前，都必须经过用户确认。

### 推荐顺序

1. 先看 `README.md`，确认目录约定和当前状态。
2. 如果用户显式指定子 skill，先按这个名字去 `active/` 查找。
3. 找到就直接调用；找不到就说明未命中，并询问是否要创建草稿。
4. 如果用户没有指定子 skill，再按任务判断是否复用已有能力。
5. 没有合适内容时，再生成新的草稿。
6. 草稿完成后，询问用户是否要载入生效池。
7. 用户确认后，再把内容移动到 `active/`。

### 额外约束

- 不要默认扫描 `drafts/`，除非任务明确要求检查草稿。
- 不要把未确认内容当作可引用、可执行、可复用的正式结论。
- 不要跳过 `README.md` 直接猜目录规则。

## 显式调用模式

当用户明确指定某个子 skill 名称时，按以下顺序处理：

1. 读取 `README.md`。
2. 在 `active/` 中查找同名子 skill。
3. 找到就直接调用。
4. 找不到就明确告诉用户未命中。
5. 如果用户希望新建，就进入显式创建流程。

### 显式调用的优先级

- 用户显式指定的子 skill 优先级最高。
- 显式指定的名字优先于自动推断。
- 如果名字和现有内容冲突，先报告冲突，再让用户决定。

## 显式创建模式

当用户明确要求创建某个子 skill 时：

1. 使用用户给出的名称创建草稿目录。
2. 生成最小的 `SKILL.md` 骨架。
3. 在草稿中写清楚名称、用途、输入、输出、步骤、验收和边界。
4. 草稿完成后，询问用户是否要载入生效池。
5. 用户确认后，再把该子 skill 放入 `active/`。
6. 如果用户不确认，就继续留在 `drafts/`。

如果 `active/` 中已经存在同名子 skill，不要默认覆盖；先询问用户是要复用旧版、覆盖旧版，还是改名后再创建。

### 名称规范

- 目录名和 `name` 字段都应使用小写连字符风格。
- 如果用户给出的名字不是合法技能名，要先规范化，再展示给用户确认。
- 对外显示可以更自然，内部路径必须稳定。

## 父 skill 的职责

父 skill 只做入口层工作，不直接承担业务实现。它的职责是：

- 判断当前任务是否应该复用已有内容。
- 决定当前内容是复用 active 还是先生成草稿。
- 控制加载边界，避免未确认内容误入正式流程。
- 输出下一步动作建议，而不是替代业务实现。
- 在需要时，引导内容从 drafts 进入 active。

父 skill 不负责：

- 直接写业务逻辑。
- 直接把草稿视为正式答案。
- 绕过用户确认把内容强行写入 active。
- 用状态目录替代实际问题分析。

## 草稿和生效两个阶段

### 草稿 `drafts/`

草稿阶段用于初始生成和快速迭代。

- 适合写第一版内容。
- 允许不完整，但必须可读。
- 目标是把想法先固定下来，便于后续确认。

### 生效 `active/`

生效阶段表示内容已经正式可用。

- 只有 `active/` 可以被正式加载。
- 进入 `active/` 的内容应当是可复用、可引用、可长期维护的版本。
- 生效后再改动，仍然要先更新草稿，再询问是否需要替换或新增。

## 最短工作流

最短工作流是：

1. 读 `.father-and-son-skills/README.md`。
2. 如果用户点名某个子 skill，先按名字查 `active/`。
3. 找到就直接调用。
4. 找不到就提示未命中，并询问是否创建。
5. 如果用户要求创建，就写入 `.father-and-son-skills/drafts/`。
6. 草稿完成后，询问用户是否要载入生效池。
7. 如果用户确认且内容通用，就替换或加入 `.father-and-son-skills/active/`。
8. 如果不通用，就继续留在 `.father-and-son-skills/drafts/`。

如果任务很简单，也不能省掉前两步。

## 输出格式

当这个 skill 被调用时，输出最好包含以下内容：

- 当前应该读哪个目录。
- 用户是否显式指定了子 skill。
- 是否存在可复用的 active 内容。
- 如果需要新内容，草稿应该放在哪里。
- 是否需要询问用户是否载入生效池。
- 如果不能直接加载，原因是什么。

如果任务涉及交付正文，输出应尽量清楚地给出：

- 结论。
- 状态判断。
- 是否建议载入生效池。
- 下一步动作。

## 失败和回退规则

- 如果 README 缺失，先停止，不要猜目录规则。
- 如果 active 中有多个相似候选，不要强行自动选一个，要列出差异再决定。
- 如果草稿不完整，继续留在 drafts 修改，不要强行进入 active。
- 如果用户不确认载入，就保持在 drafts。
- 如果载入失败，保留当前状态，不要假设已经生效。

## 常见错误

- 直接把 drafts 当正式内容使用。
- 忽略 README，凭经验猜目录规则。
- 跳过用户确认，直接把草稿放进 active。
- 让父 skill 直接承担业务实现。
- 误以为只要文件存在就能加载。
- 在没有用户确认的情况下，默认内容已经生效。
- 把复用判断和业务实现混在一起，导致边界失控。

## 一句话原则

只读 README 和 active 里的正式内容；drafts 只是中间态。用户可以显式调用指定子 skill，也可以显式要求创建指定子 skill，但最终进入 active 之前都要经过用户确认。
